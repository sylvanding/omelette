# PDF 上传到知识库全链路优化

## 要解决的问题

当前 PDF 上传到知识库可用的流程是断裂的三步手动操作：

1. 上传 PDF → 仅保存文件和提取 metadata（`status=pdf_downloaded`）
2. 用户需手动点击每篇论文的"扫描"按钮触发 OCR（`status=ocr_complete`）
3. RAG 索引 API 存在但**前端没有任何入口**调用它（`status=indexed`）

这意味着用户上传 PDF 后，知识库实际上无法用于聊天，因为论文从未被索引。

### 已知 Bug

- `upload.py` 中 `a.name` 应改为 `a["name"]`（metadata.authors 已改为 dict 列表）
- 上传对话框中 PDF 文件名过长时溢出边界

## 我们要构建什么

**全自动"上传即就绪"流水线**：用户拖入 PDF 后，系统自动完成 metadata 提取 → OCR → RAG 索引，全程提供实时进度反馈。同时引入多项体验优化提升系统的专业感。

## 为什么选择这个方案

考虑过三种方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全自动流水线 | 零操作成本，对标 Zotero/Mendeley | 需要处理后台任务和实时状态 |
| 半自动（预览确认后执行） | 给用户控制感 | 多一步操作，大部分情况不需要 |
| 手动但引导 | 实现简单 | 用户体验差，容易遗漏步骤 |

选择**全自动**：科研用户的核心需求是"上传后立刻能用"，不想关心 OCR/索引等技术细节。

## 关键决策

### 1. 后端：上传后自动触发 OCR + 索引流水线

- 在 `upload_pdfs` 返回结果后，后台异步启动每篇论文的 OCR → 索引流程
- 使用 `asyncio.create_task` 或现有的 `PipelineService` 接口
- 不阻塞上传响应——上传成功立即返回，处理在后台进行

### 2. 前端：双重进度反馈

- **纸张列表实时状态**：status badge 自动从 `pdf_downloaded` → `ocr_complete` → `indexed` 更新（轮询或 SSE）
- **全局进度通知**：顶部或底部显示"正在处理 3/5 篇文献..."的进度条

### 3. Metadata 预览

- 上传后表格中立即展示 Crossref 查询到的丰富元数据（作者、期刊、引用数）
- 论文标题优先使用提取到的标题，而非文件名

### 4. 智能标题

- 前端显示论文标题而非文件名
- 在 paper list 和 dialog 中统一用论文标题

### 5. 批量状态管理

- Papers 页面增加"全部索引"按钮，一键处理所有 `pdf_downloaded` / `ocr_complete` 状态的论文
- 上方显示统计：如"3 篇待索引 / 12 篇已就绪"

### 6. 上传查重优化

- 发现重复时直接在上传对话框中展示冲突，提供"保留旧版 / 使用新版 / 跳过"选项
- 不需要跳转到额外页面

### 7. 失败重试

- `error` 状态的论文显示"重试"按钮
- 点击后重新触发 OCR → 索引流程

## 实现分层

### 第一层：修复 Bug（立即）

1. 修复 `a.name` → `a["name"]` 回归 bug
2. 修复文件名截断问题

### 第二层：自动流水线（核心）

3. 上传后自动触发 OCR + RAG 索引（后台异步）
4. 新增后端 API：`POST /projects/{id}/papers/process` 批量处理
5. 前端轮询纸张状态或使用 SSE 推送

### 第三层：体验增强

6. 实时状态 badge 动画过渡
7. 全局处理进度通知组件
8. "全部索引"批量操作按钮
9. 失败重试按钮
10. 上传对话框中的 metadata 预览和查重展示

## 技术要点

- **后台任务**：使用 `asyncio.create_task` + `BackgroundTasks`（FastAPI 原生），不引入 Celery 等额外依赖
- **状态轮询**：前端使用 React Query 的 `refetchInterval` 在有处理中的论文时自动轮询
- **OCR + 索引**：复用现有 `OCRService` 和 `RAGService`，在后台顺序执行
